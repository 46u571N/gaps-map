SECCIÓN 01 / ITEMS 1-7

Estado y persistencia

Threads, historial de conversación, aislamiento por usuario

La API de Anthropic es stateless. Cada llamada parte de cero: el modelo no recuerda nada de la conversación anterior a menos que se lo mandes vos en el array de mensajes. Esto no es un bug, es la naturaleza del producto. La API es el motor de inferencia, no una plataforma de conversación.

Lo que en Claude Desktop se siente como continuidad nativa (la conversación que persiste, que retomás días después, que tiene un historial) es infraestructura que Anthropic construyó encima de su propia API. Cuando vas a la API directa, esa capa la tenés que construir vos.

El concepto: thread como objeto persistente

Un thread es una conversación que vive en tu base de datos con un ID propio. Cada mensaje (del usuario y del asistente) se guarda. Antes de cada nueva llamada a la API, reconstruís el historial leyendo de la DB y se lo pasás al modelo. El usuario no ve esa mecánica, pero es lo que hace que la conversación parezca persistente.

En nuestro stack este componente vive en el AgentService (Phase 02). El schema básico es directo:

SQL · Schema mínimo de threads
CREATE TABLE threads (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL,
  assistant_id  UUID NOT NULL,
  metadata      JSONB DEFAULT '{}'::jsonb,
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE messages (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  thread_id     UUID REFERENCES threads(id) ON DELETE CASCADE,
  role          TEXT NOT NULL,  -- 'user', 'assistant', 'tool'
  content       TEXT,
  tool_calls    JSONB,
  tool_call_id  TEXT,
  created_at    TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_messages_thread ON messages(thread_id, created_at);

El metadata JSONB es la clave para multi-tenancy: ahí va el cliente_id, el plan, el canal de origen. Permite filtrar threads sin tablas separadas por cliente. Es exactamente lo que el item 5 del listado describe.

La diferencia entre stateless y stateful en la práctica

El loop en PHP, simplificado, queda así:

PHP · AgentService · loop básico
public function sendMessage(string $threadId, string $userMessage): array
{
    // 1. Persistir el mensaje del usuario
    $this->messages->create([
        'thread_id' => $threadId,
        'role'      => 'user',
        'content'   => $userMessage,
    ]);

    // 2. Reconstruir el historial completo desde la DB
    $history = $this->messages
        ->where('thread_id', $threadId)
        ->orderBy('created_at')
        ->get()
        ->map(fn($m) => ['role' => $m->role, 'content' => $m->content])
        ->toArray();

    // 3. Llamar a la API con el historial completo
    $response = $this->anthropic->messages()->create([
        'model'    => 'claude-sonnet-4-6',
        'messages' => $history,
    ]);

    // 4. Persistir la respuesta
    $this->messages->create([
        'thread_id' => $threadId,
        'role'      => 'assistant',
        'content'   => $response->content[0]->text,
    ]);

    return $response;
}

Esto es exactamente lo que Backboard hace internamente. La única diferencia es que ellos te cobran por hacerlo y los datos pasan por sus servidores. Para nuestros clientes en sectores regulados, ese trade-off no es aceptable.

Estado real en Spartan IT

El schema está. El loop básico está. Lo que queda por consolidar es el ciclo completo de "retomar conversaciones viejas" como flujo de UX: hoy podés reabrir un thread por ID, pero no hay un selector de threads anteriores para el usuario, ni búsqueda dentro del historial. Para la cerealera con Luciana esto va a aparecer rápido porque ella necesita volver sobre conversaciones específicas.

SECCIÓN 02 / ITEMS 8-17

Memoria entre sesiones

La distinción más confundida del stack de IA

Antes de cualquier código, hay que dejar clara la distinción que confunde a casi todo el mundo:

Contexto de conversación es lo que está en el thread actual. Dura mientras el thread esté activo. Se carga en cada llamada como parte del array de mensajes. Es lo de la sección anterior.

Memoria persistente es un store de "hechos sobre el usuario" que vive fuera de cualquier conversación específica. Se alimenta de los threads pero existe independientemente. Cuando arranca un thread nuevo, las memorias relevantes se inyectan automáticamente en el contexto antes de que el modelo vea el primer mensaje.

Analogía: el contexto es la memoria de trabajo. La memoria persistente es lo que sabés de una persona aunque no estés hablando con ella en este momento.

Lo que la API no provee

La API no tiene ningún concepto de memoria persistente. No hay un endpoint para "guardar este hecho sobre el usuario". Tampoco hay extracción automática: el modelo no decide por sí mismo qué vale la pena recordar de una conversación. Todo eso es infraestructura que vive afuera de la API.

Cómo se construye el pipeline de memoria

Son tres componentes separados:

1. Extracción

Después de cada mensaje (o cada N mensajes, o cuando se cierra el thread), corrés un prompt aparte sobre la conversación que dice: "De este intercambio, ¿qué información estable vale la pena recordar para conversaciones futuras?". El output es un JSON estructurado.

PHP · Extractor de memoria
public function extractFacts(array $recentMessages): array
{
    $conversation = collect($recentMessages)
        ->map(fn($m) => "{$m['role']}: {$m['content']}")
        ->implode("\n");

    $response = $this->anthropic->messages()->create([
        'model'     => 'claude-haiku-4-5-20251001', // barato y rápido
        'system'    => $this->getExtractionPrompt(), // custom por asistente
        'messages'  => [['role' => 'user', 'content' => $conversation]],
    ]);

    return json_decode($response->content[0]->text, true);
}

El prompt de extracción es específico por dominio. Para la cerealera querés capturar señales comerciales (preferencias de pago, productos consultados frecuentemente). Para el laboratorio, contexto clínico recurrente. Esto es lo que el item 15 llama "custom extraction prompts" y es algo que vos podés diseñar mejor que Backboard porque conocés el dominio.

2. Almacenamiento

Una tabla simple en PostgreSQL. La columna de embedding habilita búsqueda semántica sobre las memorias (item 12).

SQL · Store de memorias de usuario
CREATE TABLE user_memories (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL,
  assistant_id UUID NOT NULL,
  fact        TEXT NOT NULL,
  category    TEXT,                  -- 'preference', 'context', 'history'
  confidence  REAL DEFAULT 1.0,
  embedding   vector(1024),       -- BGE-M3
  source_thread UUID REFERENCES threads(id),
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_memories_user ON user_memories(user_id, assistant_id);
CREATE INDEX idx_memories_emb ON user_memories
  USING hnsw (embedding vector_cosine_ops);

3. Recuperación

Antes de cada llamada al modelo, hacés una búsqueda semántica sobre las memorias del usuario con el último mensaje como query. Los top-K resultados se inyectan en el system prompt.

Lite vs. Pro: qué cambia realmente

Lo que Backboard vende como "Memory Pro" no es magia. Es un pipeline con más pasadas:

Lite — una sola pasada

Prompt simple de extracción con Haiku. Devuelve JSON. Se guarda. Listo. Útil para preferencias de UI, contexto casual, datos no críticos.

Pro — pipeline multi-paso con verificación

Tres pasadas en lugar de una:

  1. Extracción con chain-of-thought: Sonnet razonando paso a paso sobre qué información es estable vs. transitoria, qué entra en conflicto con memorias existentes.
  2. Reranking contra memorias previas: búsqueda de memorias similares del usuario. Si hay conflicto, decisión explícita: update vs. add vs. discard.
  3. Validación final: un modelo critic verifica que la extracción no introduce datos erróneos.

Para el laboratorio sí justifica Pro: un dato clínico mal extraído puede causar daño. Para la cerealera con Luciana, Lite alcanza para preferencias y patrones de consulta.

Lo que falta construir

Este es el área con más gaps abiertos en nuestro stack: 7 de 10 items en rojo. No es una crítica, es una decisión implícita de priorización. Hasta ahora ningún cliente activo lo necesita en producción. Pero el momento en que la cerealera empieza a tener volumen de queries repetitivas de Luciana, vamos a querer que el agente recuerde patrones de cuentas frecuentes sin que ella tenga que re-establecerlos cada sesión.

El componente de extracción es lo primero a implementar. El store ya está cerca (mismo schema que ya usamos para RAG). La recuperación es trivial una vez que existe el store.

SECCIÓN 03 / ITEMS 18-21

Manejo del context window

El problema silencioso que aparece tarde

Claude Opus tiene 200K tokens de ventana de contexto. Suena enorme hasta que te das cuenta de que una conversación productiva en el laboratorio puede acumular: 50 mensajes de chat + 5 informes anatomopatológicos completos + memorias inyectadas + el system prompt + las tools definidas. Sumás eso a lo largo de tres semanas y aparece el techo.

Cuando la conversación excede el límite, la API simplemente devuelve un error. No hay degradación elegante automática. Esa lógica es tuya.

Estrategias de truncación

Tres patrones, de más simple a más sofisticado:

Sliding window

Conservás el system prompt + los últimos N mensajes. Lo más viejo se descarta. Funciona para conversaciones donde el contexto reciente importa más que el histórico. Es 10 líneas de código.

Resumen progresivo

Cuando la conversación se acerca al límite, corrés un prompt aparte que resume los primeros N mensajes en un párrafo, y reemplazás esos mensajes por el resumen. Conservás la esencia pero perdés detalles. Es lo que hace ChatGPT internamente.

Resumen jerárquico

Múltiples niveles de resumen: detalle reciente + resumen de la última semana + resumen del mes anterior. Más complejo de mantener pero conserva información relevante a largo plazo.

PHP · Sliding window con margen de seguridad
public function buildContext(string $threadId, int $maxTokens = 180000): array
{
    $messages = $this->messages
        ->where('thread_id', $threadId)
        ->orderBy('created_at', 'desc')
        ->get();

    $tokenCount = 0;
    $included = [];

    foreach ($messages as $msg) {
        $estimated = $this->estimateTokens($msg->content); // ~4 chars por token
        if ($tokenCount + $estimated > $maxTokens) break;

        $tokenCount += $estimated;
        $included[] = ['role' => $msg->role, 'content' => $msg->content];
    }

    return array_reverse($included); // orden cronológico para la API
}

Para estimación rápida de tokens, la regla de "4 caracteres por token" funciona razonablemente bien para español. Si querés precisión real, Anthropic tiene un endpoint /v1/messages/count_tokens que devuelve el conteo exacto antes de mandar la llamada.

Estado real en Spartan IT

Hoy no tenemos manejo de overflow. Si una conversación supera el límite, falla. Para los volúmenes actuales (POC del laboratorio, primeras pruebas con la cerealera) no es un problema, pero es deuda técnica con vencimiento. La estrategia más simple — sliding window con resumen de la cola — es entre medio día y dos días de trabajo. Vale la pena antes de que aparezca en producción.

SECCIÓN 04 / ITEMS 22-27

Routing y multi-proveedor

Donde la decisión de Spartan diverge de Backboard a propósito

Esta sección es donde más se nota la diferencia de modelo de negocio. Backboard vende "17.000 modelos con una sola API key" como ventaja. Para sus clientes (developers individuales construyendo apps de consumo) es genuino: querés probar Llama, Mistral, Gemini sin abrir cuentas en cada proveedor.

Para nuestros clientes — laboratorios y empresas en regulados — esa amplitud es exactamente lo contrario de lo que necesitan. Cuantos menos proveedores pasen por sus datos, mejor. Pasar datos clínicos de un paciente argentino por un proxy de un SaaS canadiense que a su vez los manda a 17.000 modelos distintos en jurisdicciones impredecibles es, literalmente, un problema legal bajo Ley 25.326.

Por eso varios items de esta sección quedan marcados como "no aplica": no es que no podamos hacerlos, es que no queremos.

Lo que sí tiene sentido: routing por complejidad dentro del mismo proveedor

Anthropic tiene tres modelos vivos: Haiku, Sonnet, Opus. Cada uno con su perfil de precio y capacidad. Tiene mucho sentido usar el más barato para tareas simples y subir solo cuando el problema lo justifica.

Casos típicos de routing interno:

  • Haiku: clasificación de intent, extracción de entidades, validaciones, prompts cortos con output estructurado.
  • Sonnet: conversación general, resúmenes, búsqueda con razonamiento, generación de drafts.
  • Opus: síntesis compleja, análisis con múltiples fuentes, decisiones que requieren múltiples pasos de razonamiento.
PHP · Router básico por tarea
class ModelRouter
{
    private const ROUTES = [
        'classify'  => ['model' => 'claude-haiku-4-5-20251001'],
        'extract'   => ['model' => 'claude-haiku-4-5-20251001'],
        'chat'      => ['model' => 'claude-sonnet-4-6'],
        'summarize' => ['model' => 'claude-sonnet-4-6'],
        'analyze'   => ['model' => 'claude-opus-4-7'],
        'synthesize'=> ['model' => 'claude-opus-4-7'],
    ];

    public function resolve(string $taskType): array
    {
        return self::ROUTES[$taskType] ?? self::ROUTES['chat'];
    }
}

El ahorro real aparece a escala. Si el 70% de las queries de la cerealera son clasificaciones simples ("¿esto es consulta de saldo, factura o reclamo?"), mandarlas a Haiku en lugar de Opus reduce costos por un factor de 30. A volumen de cientos de queries por día, eso se nota en la factura mensual.

Diferenciador frente a Backboard

La restricción a un solo proveedor no es una limitación de nuestro stack, es una característica deliberada del producto. Lo que para Backboard es desventaja, para nosotros es argumento de venta: los datos del cliente no pasan por ningún intermediario fuera del que el cliente ya autorizó. Es la diferencia entre vender a developers vs. vender a hospitales.

SECCIÓN 05 / ITEMS 28-35

RAG y documentos

Donde más fuertes estamos y donde está el gap más concreto

RAG (Retrieval-Augmented Generation) es darle al modelo acceso a documentos que no estuvieron en su entrenamiento. La API no provee nada de esto: ni indexación, ni búsqueda, ni almacenamiento. Es 100% infraestructura externa.

En nuestro stack este componente está más maduro que otros. 743 PDFs indexados con pdfplumber + BGE-M3 + pgvector funcionando. Pero hay un gap concreto e importante: la búsqueda es solo semántica. Falta integrar búsqueda por keyword.

Por qué la búsqueda semántica sola no alcanza

Los embeddings funcionan bien cuando lo que se busca es significado. "El paciente tiene eritrocitosis" matchea bien con una query como "aumento de glóbulos rojos". El vector captura la relación conceptual aunque las palabras sean distintas.

Pero los embeddings fallan con identificadores literales: códigos, números de cuenta, IDs de informe, nombres propios poco comunes. Si alguien pregunta "¿qué dice el informe 4023-A?", el vector del 4023-A no tiene significado conceptual — es un string arbitrario. La búsqueda semántica va a devolver el chunk más parecido conceptualmente, que puede no contener "4023-A" en ninguna parte.

Para el laboratorio esto es crítico: códigos de protocolo, IDs de muestra, nombres específicos. Para la cerealera también: número de cliente, código de producto, número de factura.

La solución: búsqueda híbrida con RRF

Dos búsquedas en paralelo y fusión por Reciprocal Rank Fusion. La fórmula es simple: para cada chunk recuperado, score = 1/(k + rank). El chunk con mejor rank en ambas listas gana. El k estándar es 60.

SQL · Setup para búsqueda híbrida
-- Columna tsvector generada automáticamente a partir del contenido
ALTER TABLE document_chunks ADD COLUMN tsv tsvector
  GENERATED ALWAYS AS (to_tsvector('spanish', content)) STORED;

CREATE INDEX idx_chunks_tsv ON document_chunks USING gin(tsv);

-- El índice HNSW para búsqueda semántica ya existe en nuestro stack
-- CREATE INDEX idx_chunks_emb ON document_chunks USING hnsw (embedding vector_cosine_ops);
SQL · Query híbrida con RRF
WITH semantic AS (
  SELECT id, content,
         ROW_NUMBER() OVER (ORDER BY embedding <=> $1) AS rank
  FROM document_chunks
  WHERE assistant_id = $3
  ORDER BY embedding <=> $1
  LIMIT 20
),
keyword AS (
  SELECT id, content,
         ROW_NUMBER() OVER (ORDER BY ts_rank(tsv, query) DESC) AS rank
  FROM document_chunks, to_tsquery('spanish', $2) query
  WHERE assistant_id = $3 AND tsv @@ query
  ORDER BY ts_rank(tsv, query) DESC
  LIMIT 20
),
fused AS (
  SELECT
    COALESCE(s.id, k.id) AS id,
    COALESCE(s.content, k.content) AS content,
    COALESCE(1.0 / (60 + s.rank), 0) +
    COALESCE(1.0 / (60 + k.rank), 0) AS score
  FROM semantic s
  FULL OUTER JOIN keyword k ON s.id = k.id
)
SELECT * FROM fused ORDER BY score DESC LIMIT 10;

Tres parámetros ($1 embedding del query, $2 los términos para tsquery, $3 el assistant_id para multi-tenancy). El resultado es una lista combinada donde los chunks que aparecen alto en ambas búsquedas dominan, los que aparecen solo en una son secundarios, y los que no aparecen en ninguna no entran.

Detalle importante: stemming en español

to_tsvector('spanish', ...) aplica stemming: "eritrocito" matchea "eritrocitos", "factura" matchea "facturado". Esto a veces es lo que querés. Pero para códigos, IDs, nombres propios y términos técnicos exactos, conviene 'simple' que no stemiza nada. Para el laboratorio probablemente querés dos columnas tsvector (una con stemming, otra sin) y buscar en ambas.

Cuándo NO usar RAG: datos estructurados

Esta es la confusión más común y la que más cuesta cara en producción. La búsqueda semántica solo tiene sentido sobre texto libre. Sobre datos estructurados (cuentas, saldos, fechas, números de factura) hacés SQL directo.

Para responder "¿cuánto debe el cliente Pérez?" no se hace embedding de la pregunta y búsqueda vectorial. Se hace una tool que ejecuta SELECT SUM(saldo) WHERE cliente = 'Pérez'. El modelo decide cuándo llamarla y con qué argumentos, pero la respuesta sale del SQL determinístico.

Esta distinción se ve más en la sección 06 (Tool calling). Por ahora la regla:

Regla práctica

Texto libre (observaciones clínicas, descripciones, comentarios, documentación) → embedding + búsqueda híbrida. Datos estructurados (números, fechas, IDs, categorías, relaciones) → SQL directo vía tool. Si la mayoría de las columnas de una tabla son numéricas, RAG sobre esa tabla es probablemente el patrón equivocado.

Scoping de documentos: assistant-level vs thread-level

Este es el gap concreto del item 31. Hoy nuestros documentos son globales por asistente: cualquier thread del laboratorio ve todos los protocolos indexados. Bien. Pero falta el caso de "thread-level": un documento subido en una conversación específica que solo está disponible para esa conversación.

Casos donde aplica: Luciana sube un PDF con un acuerdo específico de un cliente, y no queremos que ese contenido aparezca en consultas de otros clientes. Un paciente sube su historial previo, y queremos que solo esté disponible en su thread.

El cambio es simple: un campo thread_id NULL en la tabla de documentos. Si está nulo, es global para el asistente. Si tiene valor, está scopeado a ese thread.

SQL · Two-tier document scoping
ALTER TABLE documents ADD COLUMN thread_id UUID
  REFERENCES threads(id) ON DELETE CASCADE;

-- Query que respeta el scoping:
SELECT * FROM document_chunks dc
JOIN documents d ON dc.document_id = d.id
WHERE d.assistant_id = $1
  AND (d.thread_id IS NULL OR d.thread_id = $2);
Notas al margen — opciones futuras

Caminos más sofisticados que existen pero no perseguimos hoy

Dos upgrades posibles sobre la búsqueda híbrida básica que vale tener anotados. Ninguno es urgente. La regla: implementar el camino simple, medir dónde falla, y solo entonces evaluar si la complejidad adicional se justifica.

Vectores dispersos nativos de BGE-M3

BGE-M3 no solo produce el vector denso de 1024 dimensiones que ya usamos. En la misma pasada también puede generar un vector disperso de 250.002 dimensiones — una por cada token del vocabulario del modelo. La mayoría valen cero; las que tienen valor positivo son los tokens que efectivamente aparecen en el chunk, con un peso aprendido que aproxima rareza e importancia.

Conceptualmente superior a tsvector para la parte keyword: en lugar de stemming léxico estándar, es una representación aprendida del corpus. La desventaja: pgvector clásico no soporta vectores dispersos. Se necesita pgvecto.rs (extensión separada, en evolución) o migrar a una DB vectorial dedicada como Milvus.

Cuándo justificaría el cambio: cuando midamos que la búsqueda híbrida con tsvector está dejando pasar consultas que un vector disperso aprendido capturaría. Hoy no tenemos esa medición. Sería optimización prematura.

ColBERT-style late interaction reranking

BGE-M3 también produce representaciones multi-vector: un vector por token en lugar de un solo vector por documento. Esto habilita un patrón llamado late interaction: en lugar de comparar un único vector del query contra un único vector del chunk, comparás todos los vectores token-a-token y agregás los scores. Más preciso, más caro computacionalmente.

Se usa típicamente como segunda capa de reranking sobre los top-K candidatos que devuelve la búsqueda híbrida básica. El reranker oficial de la familia BGE es bge-reranker-v2-m3, disponible bajo la misma licencia.

Cuándo justificaría el costo: en casos donde precisión importa más que latencia. Para informes anatomopatológicos del laboratorio donde una decisión clínica depende del chunk correcto, tiene sentido. Para conversación en tiempo real con Luciana, agrega latencia que probablemente no compensa.

Prioridad inmediata

Búsqueda híbrida es el cambio más alto-valor / bajo-esfuerzo del mapa. Una semana de trabajo para Tomás. Impacto directo en calidad de respuesta del laboratorio. El two-tier scoping es trivial (un día). Si tuviera que elegir el primer gap a cerrar, sería este.

SECCIÓN 06 / ITEMS 36-40

Tool calling con estado

Cómo el modelo "decide" y por qué no es magia

Tool calling es el mecanismo por el cual el modelo invoca funciones externas durante una conversación. Es lo que convierte un chatbot en un agente. Sin tools, el modelo solo puede generar texto basado en lo que tiene en el contexto. Con tools, puede consultar bases de datos, llamar APIs, ejecutar código.

Cómo "decide" el modelo qué tool llamar

El modelo no tiene capacidad de decisión propia. Vos le pasás una lista de tools con descripciones. El modelo lee las descripciones y, en función de la pregunta del usuario, elige cuál (o cuáles) tools tienen sentido. La calidad de las descripciones determina la calidad de las decisiones.

PHP · Definición de tools
$tools = [
    [
        'name' => 'search_internal_docs',
        'description' => 'Busca en la documentación interna del laboratorio: protocolos, ' .
                         'procedimientos, manuales técnicos. Usar para preguntas sobre cómo se ' .
                         'hace algo internamente o qué dice un documento específico.',
        'input_schema' => [
            'type' => 'object',
            'properties' => [
                'query' => ['type' => 'string', 'description' => 'Términos de búsqueda'],
            ],
            'required' => ['query'],
        ],
    ],
    [
        'name' => 'get_patient_results',
        'description' => 'Consulta los resultados de análisis de un paciente específico ' .
                         'por su ID. Devuelve valores cuantitativos y observaciones del ' .
                         'patólogo.',
        'input_schema' => [
            'type' => 'object',
            'properties' => [
                'patient_id' => ['type' => 'string'],
                'date_from'  => ['type' => 'string', 'format' => 'date'],
            ],
            'required' => ['patient_id'],
        ],
    ],
];

Cuando llega una pregunta como "¿cuáles fueron los resultados de Pérez del mes pasado?", el modelo lee las descripciones, identifica que get_patient_results es la que aplica, y devuelve un mensaje pidiendo invocarla con esos parámetros. Tu código ejecuta la función real (un SELECT a la DB), le devuelve el resultado al modelo, y el modelo continúa generando la respuesta con esos datos.

El loop completo con estado persistente

PHP · Loop de tool calling con persistencia
public function runWithTools(string $threadId, string $userMessage): string
{
    $this->saveMessage($threadId, 'user', $userMessage);
    $history = $this->buildContext($threadId);

    while (true) {
        $response = $this->anthropic->messages()->create([
            'model'     => 'claude-sonnet-4-6',
            'tools'     => $this->tools,
            'messages'  => $history,
        ]);

        // El modelo no necesita más tools: respuesta final
        if ($response->stop_reason === 'end_turn') {
            $text = $response->content[0]->text;
            $this->saveMessage($threadId, 'assistant', $text);
            return $text;
        }

        // El modelo pide ejecutar una o más tools
        if ($response->stop_reason === 'tool_use') {
            $toolResults = [];
            foreach ($response->content as $block) {
                if ($block->type === 'tool_use') {
                    $result = $this->dispatchTool($block->name, $block->input);
                    $toolResults[] = [
                        'type'        => 'tool_result',
                        'tool_use_id' => $block->id,
                        'content'     => json_encode($result),
                    ];
                }
            }

            // Persistir el tool_use del modelo Y los results en la DB
            $this->saveAssistantToolCall($threadId, $response->content);
            $this->saveToolResults($threadId, $toolResults);

            // Reconstruir el contexto y seguir
            $history = $this->buildContext($threadId);
        }
    }
}

El detalle clave del item 36 está en las llamadas a saveAssistantToolCall y saveToolResults: cada tool call y cada resultado se persiste en la tabla de mensajes. La próxima vez que se reconstruye el contexto, esos resultados están ahí. El modelo "recuerda" que ya consultó a Pérez la semana pasada porque está en el historial del thread.

Tools vs. Skills: no es lo mismo

Aclaración que vale repetir porque la confusión es común:

Tools (function calling)

Funciones que el modelo puede invocar vía la API. Se definen en el parámetro tools de cada llamada. El modelo decide cuándo usarlas. Esto es lo que usamos.

Skills

Carpetas con SKILL.md + recursos que productos de Anthropic (Claude Desktop, claude.ai, Claude Code, Cowork) leen proactivamente. Son una capa de orquestación del cliente, no del modelo. No están disponibles vía la API directa. Si querés equivalente funcional en tu propio agente, lo construís como tools o como contexto inyectado en el system prompt.

Casos donde el modelo no necesita tools

Si alguien sube un contrato corto (cabe en el contexto) y pide revisarlo, no hace falta tool: el contrato va directo en el mensaje del usuario, el modelo lee, responde. Tool sería necesario si el contrato es grande y hay que indexarlo (RAG), o si la revisión requiere consultar tu base de jurisprudencia interna, o ejecutar una validación contra reglas estructuradas.

Estado real en Spartan IT

El loop básico está implementado. Falta: ejecución paralela de tools (item 39) y streaming de la respuesta final (item 40). Paralelismo en PHP requiere algo más de trabajo (procesos hijos o cola). Streaming es soportado por la API de Anthropic vía Server-Sent Events; en nuestro AgentService falta integrarlo. Para Luciana viendo respuestas largas, va a importar pronto.

SECCIÓN 07 / ITEMS 41-43

Búsqueda web

Lo que no necesitamos por ahora

La API de Anthropic tiene una tool nativa de búsqueda web (web_search_20250305) que el modelo puede activar cuando detecta que necesita información fuera de su entrenamiento. Backboard lo expone como un parámetro web_search="Auto".

Para nuestros casos actuales, no aplica directamente:

  • Laboratorio: los datos relevantes son internos (protocolos, informes, resultados). La información que el modelo necesita está en los documentos indexados, no en la web.
  • Cerealera: consultas sobre cuentas, saldos, productos del cliente. Información rigurosamente interna.

Donde sí podría tener sentido en el futuro:

  • Generador de informes ejecutivos (P3 del roadmap) que necesita información de mercado actualizada.
  • Agente de investigación legal que tiene que consultar fallos recientes.
  • Asistente comercial que necesita información pública sobre clientes potenciales.

Si en el futuro lo necesitamos, agregarlo es trivial: una entrada más en el array de tools. La API maneja la búsqueda. Nosotros recibimos el resultado en el contexto.

SECCIÓN 08 / ITEMS 44-50

Multi-agente

Donde Dialectia ya validó el patrón y falta automatizarlo

Multi-agente es un agente coordinando a otros agentes especializados, cada uno con su propio system prompt, sus propias tools y potencialmente su propio modelo. Es el patrón natural cuando una tarea tiene componentes claramente separables: un investigador, un crítico, un sintetizador.

En Spartan IT ya hicimos el ejercicio manual con el cockpit de Dialectia. Probó que el patrón funciona y que distintos modelos pesan diferente en distintas tareas. Lo que falta es la versión automatizada dentro del AgentService.

La arquitectura conceptual

Tres elementos:

  1. Especialistas: agentes con system prompt focalizado y un set acotado de tools. Cada uno tiene su propio thread donde acumula su trabajo.
  2. Coordinador: agente que recibe la pregunta original, decide qué especialistas necesita, distribuye trabajo, recoge resultados, sintetiza.
  3. Memoria compartida (opcional): store de memorias del usuario al que todos los especialistas tienen acceso, para que el contexto del usuario no se duplique.

Ejecución paralela vs. secuencial

Cuando las tareas de los especialistas son independientes, deberían correr en paralelo. En PHP esto requiere procesos hijos (pcntl_fork), una cola con workers, o llamadas asíncronas con Guzzle. La opción más limpia para nuestro stack es Laravel Horizon con Redis: cada especialista se dispatch como un job, el coordinador espera con un Bus::batch().

PHP · Coordinador con batch paralelo
use Illuminate\Bus\Batch;
use Illuminate\Support\Facades\Bus;

public function runMultiAgent(string $userQuery): string
{
    // El coordinador decide qué especialistas correr
    $plan = $this->coordinator->planTasks($userQuery);

    // Dispatch en paralelo
    $batch = Bus::batch(
        collect($plan)->map(fn($task) =>
            new SpecialistAgentJob($task->specialist, $task->prompt)
        )->toArray()
    )->dispatch();

    // Esperar resultados (timeout configurable)
    $results = $this->awaitBatchResults($batch->id, timeout: 120);

    // El coordinador sintetiza
    return $this->coordinator->synthesize($userQuery, $results);
}

Cuándo NO usar multi-agente

El patrón es seductor pero caro. Cada agente es una llamada (o varias) a la API. Un sistema de 3 especialistas paralelos + coordinador es típicamente 5-8 llamadas vs. 1-2 de un agente único. Si el problema se puede resolver con un solo agente bien instruido y buen RAG, el multi-agente está sobre-engineereado.

Aplicar cuando: la tarea tiene componentes con conocimiento de dominio claramente distinto, o cuando paralelizar reduce wall-clock significativamente, o cuando el output requiere triangulación (investigador + crítico).

No aplicar cuando: la tarea es secuencial por naturaleza, o cuando un solo modelo con buenas tools ya resuelve.

Dos lecturas del item 50

El último item del listado original — "describir el sistema en lenguaje natural y que se construya solo" — es la apuesta más visible del MCP de Backboard, y la que mejor vende en un blog post. Para nuestro modelo de negocio esa lectura no aplica: la construcción de un agente para el laboratorio la hace alguien que entiende el dominio clínico, no Claude leyendo una frase. La creación es trabajo de consultoría.

Pero hay una segunda lectura del MCP que sí tiene sentido y es la más madura: como herramienta de observación y sintonía sobre los agentes que ya están en producción. La construcción es un evento puntual; la operación es la vida del sistema. Ver decisiones más abajo.

SÍNTESIS

Decisiones arquitectónicas

Por qué construimos vs. por qué no aplica

La tentación al ver una lista de 50 cosas que un producto comercial ofrece y nosotros no, es construirlas todas. La pregunta correcta no es "¿podemos hacer esto?", es "¿esto suma a nuestro modelo de negocio?". Algunas decisiones explícitas:

Multi-proveedor de modelos
No. Anthropic exclusivo. La amplitud que para Backboard es feature, para nuestros clientes regulados es problema legal. Single provider es propiedad del producto.
Embeddings locales (BGE-M3)
Sí. Crítico. Los datos no salen del ambiente del cliente para generar embeddings. Diferenciador concreto vs. cualquier solución SaaS.
DB propia (no vector DB de tercero)
Sí. pgvector en PostgreSQL. Una sola tecnología de almacenamiento, control total, sin lock-in.
Búsqueda híbrida
Sí, prioridad inmediata. PostgreSQL nativo lo permite sin infraestructura adicional.
Extracción automática de memoria
Sí, pero no aún. Esperamos al primer cliente con volumen real de queries repetitivas. Construcción a demanda.
Streaming de respuestas
Sí, pronto. UX importa para la cerealera con Luciana en producción.
MCP para auto-construcción de agentes
No. La construcción de cada agente requiere conocimiento de dominio (clínico, comercial, contable). Es trabajo de consultoría experta, no de descripción en lenguaje natural.
MCP propio read-only para operación
Sí, en evaluación. Como lente de observación y sintonía sobre agentes ya en producción. Modificaciones se hacen por el camino normal: código, revisión, deploy.
Router por complejidad (Haiku/Sonnet/Opus)
Sí, valor inmediato en costos. Implementación straightforward.
Multi-agente automatizado
Sí, pero alineado a casos concretos. Dialectia validó el patrón; el siguiente paso es automatizarlo para P3 (informes ejecutivos).

MCP read-only: el caso que sí tiene sentido

Backboard vende su MCP como herramienta de creación. Es la angulación que mejor demostración visual da: "lo construí en una frase". Pero un agente en producción se crea una vez y se sintoniza cientos de veces. La creación es el evento; la operación es la vida del sistema.

Para un equipo chico como el nuestro operando varios agentes (laboratorio, cerealera, y los que vengan), un MCP propio que exponga las operaciones del AgentService en modo solo lectura resolvería un problema real: tener un panel de control conversacional sobre el estado de los agentes en producción, sin tener que abrir el admin panel ni hacer queries manuales a la DB cada vez que algo amerita revisión.

Operaciones a exponer (todas read-only)

Flujo típico de sintonía

Ejemplo de uso

Vos en Claude Code, MCP de operación conectado:

"Mostrame las últimas 10 queries que devolvieron respuestas muy cortas en el agente del laboratorio."

Claude consulta el MCP, identifica patrones (por ejemplo: todas son consultas con códigos de protocolo). Le pedís que pruebe una de esas queries contra el RAG para ver qué chunks recupera. Detectás que la búsqueda semántica no encuentra los códigos. Anotás: hay que activar búsqueda híbrida. Las modificaciones reales se hacen en código, no por el MCP.

Por qué read-only es la postura correcta

Dar write access a un MCP sobre una DB de producción es darle a Claude — y a quien tenga acceso al cliente — capacidad de modificar el comportamiento de agentes que atienden clientes reales. Para un caso de consumo (un developer experimentando con su propia app), el riesgo es acotado. Para nuestros clientes en sectores regulados, no.

Los cambios sobre prompts, tools, modelos, o cualquier configuración de un agente siguen el camino normal: branch, PR, revisión, deploy. El MCP nos da visibilidad y diagnóstico; la modificación está fuera de su alcance. Es la misma disciplina que aplicamos para cualquier cambio en producción.

Nota práctica: implementar el MCP es relativamente barato (las 8 operaciones de arriba son endpoints HTTP envueltos en formato MCP). El valor aparece cuando empieza a haber suficiente actividad como para que el debugging manual sea fricción. Hoy con dos clientes y volumen bajo, no es urgente. Cuando el laboratorio entre en producción real, va a empezar a ser útil.
